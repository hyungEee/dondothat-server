# dondothat-server
안녕하세요! 최지형입니다. Don do that 팀프로젝트 - 백엔드 파트에서 제가 맡은 역할은 다음과 같습니다.
- LLM을 활용한 소비내역 카테고리 분류 시스템
- LLM을 활용한 사용자 맞춤 챌린지 추천 시스템
- 저금통 API 설계
- 챌린지 성공/실패 판정 로직 설계
</br>
해당 프로젝트에서 진행한 내용 + 개인적으로 고도화한 내용까지 포함해 작성해보았습니다.

## LLM 소비 카테고리 분류 시스템
<img width="1913" height="1079" alt="image" src="https://github.com/user-attachments/assets/2f9e945b-0f55-46fa-b706-f919d3a740a1" />
입출금 계좌 데이터 연동 시 카드와 달리 업장 카테고리 정보가 제공되지 않아 자체 카테고리 분류 로직을 설계했습니다. </br>
(외부 API의 경우 카테고리가 저희가 원하는 분류와 다른 기준으로 분류되어 있거나, 항목이 비어있는 경우가 부지기수였습니다.) </br>
사전 기반 규칙으로만 접근하기에는 고유명사로 이루어진 상호명도 많았기에 한계가 있어 LLM을 적용했습니다. </br>
주요 키워드(ex.프랜차이즈 상호명)로 1차 필터링을 수행하고, 남은 항목만 LLM이 판단하는 하이브리드 방식을 고안했습니다. </br>
이를 통해 분류 정확성을 높이고 호출 비용을 줄일 수 있습니다. </br></br>

<img width="1919" height="1070" alt="image" src="https://github.com/user-attachments/assets/5f4b84c3-bc50-4d50-b5a1-605da59e3451" />
<img width="1914" height="1075" alt="image" src="https://github.com/user-attachments/assets/8c9fb047-4fab-4047-8b25-a3783c953a9b" />
한번에 많은 양의 데이터 입력 시에는 비동기 호출을 적용하여 응답 속도를 개선하고자 했습니다. </br>
배치와 세마포어 사이즈에 따른 테스트를 진행하여 최적의 값을 적용했습니다. </br> 
이때, 호출 비용적인 측면을 고려하여 배치사이즈를 한자리수까지 과하게 줄이지 않았고 16으로 설정했습니다. </br>
이에따라 평균 응답 속도를 약 65% 단축시킬 수 있었습니다. </br></br>

## LLM 사용자 맞춤 챌린지 추천 시스템
<img width="1916" height="1075" alt="image" src="https://github.com/user-attachments/assets/be502b7f-d031-4376-ae8a-ee45bb394839" />
사용자의 소비 데이터와 저희 서비스의 챌린지 데이터를 LLM에 전달해, 사용자의 소비습관에 적절한 챌린지를 추천해주는 기능입니다.</br>
사치성 소비나 새로운 지출 항목에는 가산점을 부여하게끔 설정했습니다.</br>
정해진 챌린지 내에서 정확히 3개의 챌린지만 추천해야하기 때문에, output schema를 적용해 출력형식을 강제했습니다.</br></br>
LLM을 활용한 위 두가지 기능은 모두, 응답 받은 뒤 유효성 검사 및 비정상 출력시 기본값 폴백을 적용하여 안정성을 보장했습니다.</br>

## 저금통 API 설계
<img width="1940" height="1096" alt="image" src="https://github.com/user-attachments/assets/879b4b28-f9cd-49fd-9b8a-1ea395272e90" />
사용자가 챌린지 성공시 아낀 금액을 저금통 계좌로 이체함으로써, 절약 현황을 명시적으로 볼 수 있게 해주는 기능입니다.</br>
시중에 실제 입출금이 가능한 API가 없기에, 실제 이체가 되지는 않지만 최대한 비슷하게 구현하려 노력하였습니다.</br>
서버에서는 트랜잭션과 롤백 설정, sql에서는 원자적 업데이트문을 통해 동시성을 보장하였습니다.</br>
또한 중복 클릭 등 동일 요청 반복 시 결과가 한번만 반영되게 하기 위하여 원장 테이블을 설계해 멱등성을 보장했습니다.</br>
해당 테이블에서는 user_challenge_id를 유니크키로 지정하여 한 유저 챌린지 당 한번의 저축만 가능합니다.</br></br>

## 챌린지 성공/실패 판정 로직 설계
<img width="2560" height="1600" alt="image" src="https://github.com/user-attachments/assets/1e7a95d6-81e5-4828-a1d0-185fbb255a1c" />
매일 밤 자정, 챌린지를 진행중인 모든 사용자의 당일 소비내역을 확인한 뒤 금지된 항목의 소비가 있는지 판단하고, 
항목이 있을 시엔 실패, 항목이 없을 시엔 성공, 마지막날일 경우에는 최종성공 처리를 하는 스케줄러를 구현했습니다.
세심하게 고려한 부분은 @Scheduled와 @Transactional을 한번에 설정하지 않고 클래스를 분리한 점입니다.
@Transcational은 스프링 내부에서 프록시로 처리됩니다. @Schedueled는 내부적으로 직접 메서드를 호출하는게 아니라,
스프링이 스케줄러 빈을 실행하면서 메서드를 프록시 없이 직접 호출해 버린다면 트랜잭션이 적용되지 않거나 꼬일 수 있기 때문입니다.
따라서 구조를 분리하여 트랜잭션의 안전성을 보장했습니다.

위 기능은 배치 시스템으로 고도화 후, 내용을 더 업데이트할 예정입니다.
